## 바이브 코딩만으로 충분하지 않을 때

Module 2에서 바이브 코딩으로 앱을 만들어봤습니다. 잘 되죠?\
그런데 만들 앱이 좀 더 복잡해지면 이런 문제가 생깁니다:

| 상황 | 문제 |
| --- | --- |
| 기능이 10개 이상 | 채팅으로 하나씩 요청하면 앞에 만든 것이 깨질 수 있음 |
| 여러 명이 함께 작업 | "뭘 만들지" 합의가 안 된 상태로 진행하면 혼란 |
| 나중에 수정할 때 | 처음 요구사항이 뭐였는지 기억이 안 남 |

이럴 때 **Spec**을 먼저 작성하면, AI가 전체 그림을 보고 체계적으로 만들어줍니다.

{% hint style="info" %}
**편의점 비유**: 바이브 코딩 = 인테리어 업자에게 말로 하나씩 지시하는 것\
Spec = 인테리어 **설계도면**을 먼저 그려서 전달하는 것\
\
어느 쪽이든 결과물은 나오지만, 복잡한 공사일수록 도면이 있는 쪽이 실수가 적습니다.
{% endhint %}

***

## 바이브 코딩 vs Spec, 언제 뭘 쓸까?

| | 바이브 코딩 | Spec |
| --- | --- | --- |
| **적합한 경우** | 간단한 앱, 빠른 프로토타입 | 기능이 많은 앱, 팀 프로젝트 |
| **장점** | 빠르고 직관적 | 체계적이고 누락 방지 |
| **단점** | 복잡해지면 앞뒤가 안 맞을 수 있음 | 초기 작성에 시간 소요 |
| **오늘 기준** | Module 2에서 한 것 | 지금 할 것 |

{% hint style="success" %}
오늘 워크샵에서는 Spec의 **"요구사항 정리 -> AI가 설계도 생성"** 단계까지만 체험합니다.\
Spec에서 코드를 자동 생성하는 것은 시간 관계상 생략하지만, 이 과정만 경험해도 Spec의 가치를 충분히 느끼실 수 있습니다.
{% endhint %}
